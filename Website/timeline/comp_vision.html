Computer Vision Model

Due to the nature of machine learning, we knew that developing our own models without a good dataset would be a challenge. As such, we initially aimed to develop a Python program capable of running detections with any computer vision model. This approach allowed us to spend time finding adequate models to deploy once the program was complete.

The application proved difficult to develop due to the various frameworks and formats of CV models. Pursuing it further no longer made sense. However, we discovered two datasets: one for identifying tomatoes and their ripeness level, and one for identifying diseases in tomato leaves. This discovery solved the problem of requiring a good dataset for model development. The new objective became creating these two models using YOLOv8 and developing a program to run detections using any YOLOv8 model.

To create the tomato image segmentation model, we first parsed the dataset into a format suitable for training. After this, we used the parsed dataset to train an image segmentation model. Lastly, we developed a Python script to run this model on a given image. This script could also be imported and used as a detection function. Multiple iterations of this script were developed, which we will explore further later.

Developing the leaf disease object detection model posed several challenges. Parsing the data was difficult due to its lack of normalization and the sheer volume of data points. Additionally, the dataset quality was lower than expected, with many grainy or out-of-focus images. Identifying slightly different leaves in a field of leaves is a more complex task than identifying red balls in a green field. To address these issues, we created a more complex model with a larger matrix, which was harder to train but more effective at achieving our goal.

The Python program was developed alongside the models to meet evolving demands. Initially, we allowed for different models to be used. We then altered how images were saved. Subsequently, we made it so that a .json file was created with the detections. Later, we implemented threading to make the process non-blocking and compatible with the database in use. The final change included adding the model name in the .json file.
